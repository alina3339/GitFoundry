The owner's manual.

Plain English, written specifically for your site; not generic documentation. Covers every routine task you might want to do, with screenshots and exact paths. Below is a real excerpt from a recent build (sector-anonymised).

1.0 · owner's manual Changing the headline on the home page 2 minutes

What you'll do

Replace the text inside the big bold sentence at the top of the home page.

Steps

  1. Go to github.com/your-account/your-site
  2. Click on index.html
  3. Click the pencil icon (top right of the file view); this opens the edit screen
  4. Use Ctrl+F (or Cmd+F on Mac) and search for hero-title
  5. You'll see your current headline between two <h1> tags. Replace the text; leave the tags alone
  6. Scroll to the bottom of the page, write a short note like "Updated headline", and click Commit changes

What happens next

GitHub starts deploying the change immediately. Your live site updates in about 30 seconds. Hard-refresh your browser (Ctrl+Shift+R) to see the new headline.

If it doesn't work

Check the Actions tab in your repo. If there's a red X next to the most recent run, something in the file got broken; usually a missing < or >. Revert the change by clicking the commit and choosing "Revert".

Page §1 · Routine updates
2.0 · owner's manual Swapping the photo on the home page 5 minutes

Before you start

Make sure your replacement photo is:

  • JPG or WebP format (not HEIC, not raw)
  • At least 1600px wide for the hero, 800px wide for everywhere else
  • Compressed; use squoosh.app if you're not sure how

Steps

  1. In your repo, navigate to /assets/images/
  2. Click Add file → Upload files
  3. Drop your new image in. Name it descriptively; for example shop-front-2026.jpg
  4. Commit the upload
  5. Now open index.html and search for the old filename (look for img src=)
  6. Replace just the filename portion. Save and commit.

One thing to watch

Keep both old and new images in the repo until you've checked the site looks right. If something breaks, you can revert with one click.

The emergency runbook.

One page. Designed for stressful moments; when your site looks broken and you need to know what to check first. The real one is printable and lives on your phone.

runbook · emergency If your site looks broken read top to bottom

Step 1; Is it actually broken?

Open isitdownrightnow.com and check your domain. If they show the site as up, the problem is on your device; try a different browser or your phone's mobile data.

Step 2; Is GitHub Pages down?

Check githubstatus.com. If "GitHub Pages" shows red or yellow, the issue is on GitHub's side and they're already on it. Most outages resolve within an hour. Nothing to do but wait.

Step 3; Did a recent edit break it?

In your repo, click the Actions tab. A red X on the most recent run means your last commit broke the build. Click the commit, then click Revert. Site will be back in about 60 seconds.

Step 4; None of the above?

Email me: info@gitfoundry.co.uk. Subject line: "your-domain.co.uk; site down". I'll look at it within the hour during UK business hours, same evening otherwise. (90-day support window applies; after that, a small fix is £25.)

Item 06 of 07 · sample below

The migration playbook.

If you ever need to move your site off GitHub Pages; to Cloudflare Pages, Netlify, GitLab Pages, or anywhere else; the playbook tells you exactly how. Under an hour. Below is the short version.

playbook · migration Moving to Cloudflare Pages ≈45 minutes

Why you might want to

Cloudflare Pages is slightly faster on average (edge network) and has more generous bandwidth limits. Free, same as GitHub Pages. The trade-off: it's another service to log in to.

What you'll need

  • A Cloudflare account (free at cloudflare.com)
  • Your domain; same domain as before
  • Your GitHub repository; no changes needed
  • About 45 minutes

Steps

  1. In Cloudflare, go to Workers & Pages → Create application → Pages → Connect to Git
  2. Authorise Cloudflare to read your GitHub repos
  3. Pick your site's repository
  4. Build settings: leave everything blank (this is a static site, no build needed). Output directory: /
  5. Click Save and Deploy. Cloudflare gives you a temporary URL like your-site.pages.dev. Test it.
  6. In Cloudflare Pages → your project → Custom domains, add your real domain
  7. Cloudflare will give you DNS records. If your domain is already on Cloudflare, it's one click. If it's elsewhere, copy the records to your registrar.
  8. Once the domain propagates (usually 5 to 30 minutes), your site is live on Cloudflare
  9. In GitHub, go to your repo → Settings → Pages → set Source to None. This stops GitHub from serving the site.

If it goes wrong

You haven't lost anything. The repo is unchanged. You can put GitHub Pages back on by reversing the last step.
